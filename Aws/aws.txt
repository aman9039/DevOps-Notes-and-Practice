1. What is Elastic Load Balancer (ELB)?
AWS Elastic Load Balancer automatically distributes incoming traffic across multiple EC2 instances. It improves high availability, scalability, and fault tolerance.

1.Why do we need a load balancer ?
Answer : 
We use a Load Balancer to distribute incoming traffic across multiple servers so that no single server becomes overloaded. It improves availability, scalability, and fault tolerance.

Imagine you have only one EC2 server.
Users --> EC2-1

Initially 100 users visit the website and everything works fine.
After some time, 10,000 users visit at the same time.

Problems:

CPU becomes high.
Memory usage increases.
Application becomes slow.
Server may crash.
Website becomes unavailable.

This is called a Single Point of Failure.

Now with Load Balancer:
=======================

                 EC2-1
Users --> ELB --> EC2-2
                 EC2-3

The Load Balancer distributes requests among all servers.

Benefits:

✅ High Availability
✅ Better Performance
✅ Fault Tolerance
✅ Supports Auto Scaling
✅ No single server overload

2. How does traffic flow?
Client sends request to ELB DNS name. Listener receives traffic and forwards it to the target group. Target group selects a healthy EC2 instance and sends the request to it. The response returns back through the Load Balancer to the client.

Step 1: User opens website
User types: www.company.com

Step 2: DNS resolves domain name

DNS converts:
www.company.com  ---> into --> my-alb-123456.ap-south-1.elb.amazonaws.com

which is the ELB DNS name.
Traffic now goes to the Load Balancer

Step 3: ELB receives request
Example:

GET /
or
GET /api/users

ELB receives the request on:
HTTP 80
HTTPS 443

Step 4: Listener receives traffic
Listener is like a gatekeeper.

Example:
If request comes on port 80    |  ex1. /images/* -> Image Target Group
Forward to Target Group        |  ex2. -/api/*    -> API Target Group

or
If request comes on port 443
Forward to Target Group

Listener rules decide where traffic should go.

Step 5: Target Group selects healthy server
Example:

Target Group:
EC2-1 Healthy
EC2-2 Healthy
EC2-3 Unhealthy

Load Balancer chooses one healthy instance.

Step 6: EC2 processes request
Example:

nginx receives request
Application processes it
Database query runs

Response is generated.

Step 7: Response returns to user
User
 ↑
EC2
 ↑
Target Group
 ↑
Listener
 ↑
ELB

Complete flow:
==============

Client
  ↓
DNS
  ↓
ELB
  ↓
Listener
  ↓
Target Group
  ↓
Healthy EC2
  ↓
Response

3. What happens if an instance fails?
ELB performs periodic health checks on EC2 instances. If an instance fails health checks, ELB stops sending traffic to it and routes requests only to healthy instances.

Suppose:
EC2-1 Healthy
EC2-2 Healthy
EC2-3 Crashed

Without Load Balancer:
=====================
Users -> EC2-3
Website becomes unavailable.

With Load Balancer:
==================
ELB continuously checks server health.

Example:
Every 30 seconds ELB sends:

GET /
If server returns: HTTP 200 OK
instance is healthy.

If server returns:
500,502,503,Timeout,Connection refused

instance becomes unhealthy.
ELB automatically removes failed server.

Before:
ELB
 |
 |-- EC2-1
 |-- EC2-2
 |-- EC2-3

After failure:
ELB
 |
 |-- EC2-1
 |-- EC2-2

Users continue using application without noticing.

When EC2-3 becomes healthy again:
systemctl start nginx
Health checks pass.
ELB automatically adds it back.

4. How do you troubleshoot ELB issues?
I check application status, listening ports, logs, security groups, target group health checks, listener rules, and target registration.

Check 1: Is application running?
systemctl status nginx/httpd/myapp

Check 2: Is application listening on correct port?
ss -tunlp or netstat -tunlp

Example:
LISTEN 0 511 0.0.0.0:80

Good.
If application listens on: 8080
but target group checks: 80
health check fails.

Check 3: Test application locally
curl localhost or curl localhost:80
Expected:
HTTP/1.1 200 OK

Check 4: Check logs
Nginx:
tail -f /var/log/nginx/error.log

Apache:
tail -f /var/log/httpd/error_log

System logs:
journalctl -xe

Check 5: Security Group
Load Balancer SG:
Allow: 80,443

EC2 Security Group:
Allow: 80 from Load Balancer Security Group

Many candidates forget this.

Check 6: Health Check Path
Wrong: /health
Application only serves: /
Result: 404 Not Found

Target becomes unhealthy.

Check 7: Listener Rule
Verify:

80 -> Target Group
443 -> Target Group

Check 8: Target Registration
Verify instance exists in target group.

5. How does ELB integrate with Auto Scaling?
Auto Scaling automatically adds newly created EC2 instances to the target group and removes terminated instances. ELB starts sending traffic only after health checks pass.

Suppose you have:

2 EC2 instances

Traffic suddenly increases.
CPU becomes:
85%,90%,95%

Auto Scaling policy says:
If CPU > 70%
Launch new instance

AWS creates: EC2-3
Auto Scaling automatically registers EC2-3 into Target Group.

Now traffic becomes:
ELB
 |
 |-- EC2-1
 |-- EC2-2
 |-- EC2-3

No manual work required.
Traffic decreases.
CPU becomes: 10%,15%

Auto Scaling policy says:
If CPU < 20%
Terminate one instance
Auto Scaling removes EC2-3.

ELB automatically stops sending traffic to it.

6. How does ELB work with Multiple Availability Zones?
ELB can distribute traffic across multiple Availability Zones. If one AZ fails, traffic automatically goes to healthy instances in another AZ, ensuring high availability and fault tolerance.

Good design:

ap-south-1a
 |
 |-- EC2-1

ap-south-1b
 |
 |-- EC2-2

ELB sits above both AZs.

             ELB
            /   \
         AZ-1   AZ-2
         EC2-1  EC2-2

If AZ-1 fails:
ELB
 |
 |-- EC2-2

Application continues working.

This is called High Availability.
